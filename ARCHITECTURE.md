# Архитектура Frontend-приложения

Руководство описывает архитектурный подход, принятый в проекте: Feature-Sliced Design (FSD) — методология организации кода, ориентированная на предметную область и масштабируемость.

---

## Обзор

[Feature-Sliced Design](https://feature-sliced.design) — архитектурная методология для фронтенд-приложений. Код организован по трем измерениям, образующим иерархию:

```
Слои → Слайсы → Сегменты
```

Каждый уровень имеет строгую зону ответственности. Методология не диктует технологии — только принципы организации кода.

---

## Слои

Слои — верхний уровень иерархии. Они стандартизированы во всех FSD-проектах, а их порядок строго задан сверху вниз:

```
app
pages
widgets
features
entities
shared
```

> `processes` — устаревший слой, не используется.

### Правило импортов

Модули на одном слое могут импортировать только из слоев **строго ниже**. Импорты вверх или вбок на том же слое запрещены.

```
app       ← может импортировать из всех слоев ниже
pages     ← может импортировать из widgets, features, entities, shared
widgets   ← может импортировать из features, entities, shared
features  ← может импортировать из entities, shared
entities  ← может импортировать из shared
shared    ← не импортирует из других слоев
```

Это правило гарантирует, что изменения в одной части системы не порождают непредвиденных последствий в другой.

### Описание слоев

| Слой       | Описание                                                                                                          |
| ---------- | ----------------------------------------------------------------------------------------------------------------- |
| `app`      | Все, благодаря чему приложение запускается — роутинг, точки входа, глобальные стили, провайдеры и т.д.            |
| `pages`    | Полные страницы или большие части страницы при вложенном роутинге                                                 |
| `widgets`  | Большие самодостаточные куски функциональности или интерфейса, обычно реализующие целый пользовательский сценарий |
| `features` | Повторно используемые реализации целых фич продукта, то есть действий, приносящих бизнес-ценность пользователю    |
| `entities` | Бизнес-сущности, с которыми работает проект, например `user` или `product`                                        |
| `shared`   | Переиспользуемый код, особенно когда он отделен от специфики проекта/бизнеса, хотя это не обязательно             |

> **Важно:** Слои `app` и `shared`, в отличие от других слоев, не имеют слайсов и состоят из сегментов напрямую.

## Слайсы

Слайсы делят слой по предметной области. Их названия и количество произвольны — главное, чтобы они отражали бизнес-домен, а не техническое устройство.

```
entities/
├── branch/
├── invoice/
└── partner/
```

Слайсы группируют тесно связанный по смыслу код, что упрощает навигацию по проекту. При этом слайсы изолированы друг от друга: один слайс не может импортировать из другого слайса того же слоя.

```ts
// ❌ Запрещено — cross-import между слайсами одного слоя
import type { Branch } from '~/entities/branch'

// это находится внутри entities/partner — нарушение правила
```

### Cross-imports между слайсами

В редких случаях слайсы внутри `entities` все же могут быть логически зависимы друг от друга. Для таких ситуаций FSD предусматривает специальный публичный API — нотацию `@x`:

```ts
// entities/branch/index.ts
export type { Branch } from './model/branch.types.ts'

// entities/partner/@x/branch/index.ts  ← явный cross-import через @x
export type { PartnerBranch } from './@x/branch/model/partner-branch.types.ts'
```

`@x`-экспорты — это явный контракт: слайс сознательно открывает часть своего API для конкретного соседа. Использовать эту возможность стоит осознанно и минимально.

## Сегменты

Сегменты — нижний уровень иерархии. Они группируют файлы внутри слайса (или слоя) по **назначению кода**, а не по типу файла.

### Стандартные сегменты

| Сегмент  | Назначение                                                                   |
| -------- | ---------------------------------------------------------------------------- |
| `ui`     | Все, что связано с отображением: UI-компоненты, форматтеры дат, стили и т.д. |
| `api`    | Взаимодействие с бэкендом: функции запросов, типы данных, мапперы            |
| `model`  | Модель данных: схемы валидации, интерфейсы, хранилища и бизнес-логика        |
| `lib`    | Библиотечный код, который нужен другим модулям этого слайса                  |
| `config` | Файлы конфигурации и фиче-флаги                                              |

Обычно этих сегментов достаточно для большинства слоев. Собственные сегменты создают, как правило, только в `app` и `shared`.

### `model` vs `lib` — частый источник путаницы

Граница между `model` и `lib` не всегда очевидна. Ключевой вопрос: **к чему относится код — к модели данных или к вспомогательной инфраструктуре?**

**`model`** содержит:

- Интерфейсы и типы, описывающие бизнес-сущность
- Схемы валидации (Valibot)
- Хранилища (React Context, Zustand)
- Хуки, работающие с контекстом или хранилищем

**`lib`** содержит:

- Утилиты общего назначения (`formatters`, `parsers`)
- Вспомогательные хуки без привязки к бизнес-логике (`useMeasure`, `useMediaQuery`)
- Гибкие типы утилитарного характера, не привязанные к конкретной сущности

#### Куда помещать типы

Тип идет в `model`, если он описывает **модель данных сущности**. Тип идет в `lib`, если он **утилитарный** — гибкий, не привязанный к конкретному слайсу, применимый в разных контекстах.

## Именование файлов

### Привязка к сущности

Внутри слайса файлы должны быть явно привязаны к сущности через префикс. Общие имена без префикса допускаются только в `app` и `shared`.

```
// ❌ Неправильно — общее название не привязывает логику к сущности
entities/invoice/api/
├── client.ts
└── queries.ts

// ✅ Правильно — название файла отражает принадлежность к сущности
entities/invoice/api/
├── invoice.client.ts
└── invoice.queries.ts
```

Когда в проекте десятки слайсов, открытый `queries.ts` в редакторе ничего не говорит о контексте. `invoice.queries.ts` — говорит сразу.

### Префикс — имя слайса, а не домена во множественном числе

Префикс файла всегда совпадает с именем слайса — в той форме, в которой слайс назван. Если слайс называется `invoice`, префикс файла — `invoice`, а не `invoices`.

Множественное число допустимо внутри файла — в именах переменных, функций и типов — если это семантически оправдано. Но имя файла всегда следует имени слайса.

### Разделение типов

Единый файл `types.ts` быстро превращается в свалку. Типы следует разделять по смыслу:

| Файл                    | Содержимое                                                          |
| ----------------------- | ------------------------------------------------------------------- |
| `{entity}.params.ts`    | Типы параметров запросов: пагинация, сортировка, фильтры            |
| `{entity}.responses.ts` | Типы ответов API: структуры данных, возвращаемых бэкендом           |
| `{entity}.types.ts`     | Общие типы сущности: интерфейсы, перечисления, вспомогательные типы |

### Имена сегментов и файлов — цель, а не суть

Частая ошибка — называть файлы и сегменты по тому, **что** в них лежит, а не **зачем** этот код написан.

```
// ❌ Неправильно — названия описывают тип содержимого
shared/lib/
└── utils.ts   ← все подряд в одном файле

// ✅ Правильно — каждый файл описывает цель
shared/lib/
├── formatters.ts   ← функции форматирования данных
└── parsers.ts      ← функции разбора строк и структур
```

Та же логика применима к сегментам. Названия `components`, `hooks`, `modals` описывают **что это**, но не **зачем**. В итоге в такие папки попадает несвязанный код, который сложно найти, сложно тестировать и сложно рефакторить.

> Название должно отвечать на вопрос «зачем существует этот код?», а не «что технически в нем находится?»

```
// ❌ Называем по сути
shared/lib/
├── hooks/       ← что это? хуки. но зачем?
└── utils/       ← что это? утилиты. но зачем?

// ✅ Называем по цели
shared/lib/
├── formatters/  ← форматируют данные для отображения
└── parsers/     ← разбирают внешние данные во внутренние структуры
```

## Публичный API

Каждый слайс и каждый сегмент общается с внешним миром через явный публичный API — файл `index.ts` на уровне слайса.

```
entities/branch/
├── api/
├── model/
└── index.ts  ← единственная точка входа
```

```ts
// entities/branch/index.ts
export { ... } from './api/branch.client.ts'
export { ... } from './api/branch.queries.ts'
export type { ... } from './model/branch.params.ts'
export type { ... } from './model/branch.responses.ts'
export { ... } from './model/branch.types.ts'
```

Импортировать напрямую из внутренних файлов слайса запрещено:

```ts
// ✅ Правильно
import type { Branch } from '~/entities/branch'
// ❌ Запрещено
import type { Branch } from '~/entities/branch/model/branch.types.ts'
```

## Внутренние импорты

Внутри `src/shared/ui/<group>/<component>/` используйте относительные импорты для файлов того же компонента. Компонент сам обращается к своим сегментам напрямую, без собственного barrel-файла:

```ts
import { buttonVariants } from '../lib/button.variants'
import type { ButtonProps } from '../model/button.props'
```

Если компонент зависит от конфигурации, утилит или соседних компонентов, используйте ближайший публичный API нужной области:

```ts
import { MEDIA_QUERIES } from '~/shared/config'
import { classNames, useMediaQuery } from '~/shared/lib'

import { Button, buttonVariants } from '../../button'
```

Для `src/shared/ui/core/<component>/` это означает:

- общая конфигурация — через `~/shared/config`;
- общие utilities/types/hooks — через `~/shared/lib`;
- файлы своего компонента — через `../config`, `../lib`, `../model`, `../ui`;
- соседние core-компоненты — через `../../<component>`, то есть через public API компонента;
- Storybook stories и docs-примеры могут импортировать из агрегирующего barrel `~/shared/ui/core`, чтобы показывать потребительский сценарий.

`~/shared/ui/core` и `~/shared/ui/addons` — агрегирующие barrel-точки для внешнего API, Storybook/docs-примеров и групповых entrypoint-файлов. Runtime-код внутри отдельного core-компонента не должен импортировать соседние компоненты из `~/shared/ui/core`, чтобы не создавать лишнюю связанность с полным групповым barrel.

## Преимущества

| Свойство                                              | Описание                                                                                                                                                                                                             |
| ----------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Однородность**                                      | Поскольку структура стандартизирована, проекты становятся более единообразными, что облегчает набор новых участников в команду                                                                                       |
| **Устойчивость к изменениям и рефакторингу**          | Модуль на одном слое не может использовать другие модули на том же слое или слоях выше. Это позволяет вам вносить изолированные правки без непредвиденных последствий для остальной части приложения                 |
| **Контролируемое переиспользование логики**           | В зависимости от уровня вы можете сделать код либо очень переиспользуемым, либо очень локальным. Это сохраняет баланс между соблюдением принципа DRY и практичностью                                                 |
| **Ориентация на потребности бизнеса и пользователей** | Приложение разделено на бизнес-домены, и при именовании поощряется использование терминологии бизнеса, чтобы вы могли делать полезную работу в продукте, не вникая полностью во все другие несвязанные части проекта |

## Постепенное внедрение

Если у вас есть существующая кодовая база, которую вы хотите перенести на FSD, мы предлагаем следующую стратегию. На нашем собственном опыте миграции она хорошо себя зарекомендовала.

1. Начните постепенно формировать слои `app` и `shared`, чтобы создать фундамент.
2. Раскидайте весь существующий интерфейсный код по виджетам и страницам, даже если у них пока что есть зависимости, нарушающие правила FSD.
3. Постепенно исправляйте нарушения правил на импорты, а по ходу извлекайте сущности и, возможно, фичи.

> Рекомендуется воздержаться от добавления новых крупных сущностей во время рефакторинга, а также рефакторинга по частям.
